# GCP 수집 누락 데이터 업로드 배치 - 프로젝트 보고서

---

## 1. 확산 가능성

### 사업장 단위 확산
- 현재 시스템은 `corp_cd` 변수 하나로 사업장을 구분하는 구조로 설계되어 있음
- 동일한 파이프라인을 다른 사업장에 적용할 때 `corp_cd` 값 변경만으로 즉시 적용 가능
- Mage.AI의 파이프라인 변수(variable) 기능을 통해 사업장별 파라미터를 분리 관리
- 신규 사업장 추가 시 `comm_df.clct_table_mt` 테이블에 비교 대상 테이블 정보만 등록하면 자동으로 모니터링 대상에 포함됨

### 데이터 종류 단위 확산
- 현재 PLC 태그 데이터, ID Tracking 데이터 2종으로 구분되어 있으나
- `bq_dataset_nm`, `bq_table_nm` 패턴 기반으로 분기 처리하는 구조이므로
- 신규 데이터 종류 추가 시 분기 조건과 쿼리 함수만 추가하면 확장 가능
- `comm_df.clct_table_mt` 메타 테이블에 신규 항목 등록만으로 모니터링 범위 확장

### 파이프라인 패턴 재사용
- BQ/PG 건수 비교 → 시간별 세분화 → 누락 데이터 재업로드 → GCS 백업의 4단계 패턴은
- 동일한 아키텍처(PostgreSQL + BigQuery + GCS)를 사용하는 타 프로젝트에 그대로 적용 가능
- Mage.AI 블록 구조로 모듈화되어 있어 필요한 블록만 선택적으로 재사용 가능
- GCS 백업 프로시저(`export_2_gcs`)는 테이블명, 파티션 컬럼, 날짜 범위만 변경하면 범용 활용 가능

### 모니터링 자동화 패턴으로 확산
- 데이터 수집 파이프라인에서 발생하는 누락/오류를 자동으로 감지하고 복구하는 패턴은
- 유사한 ETL 환경(Edge → 중앙 DB → 클라우드 DW)에서 공통 문제이므로
- 본 프로젝트의 설계 방식을 조직 내 표준 패턴으로 정립하여 타 팀/시스템으로 확산 가능

## 2. AI 활용 증빙 설명

### 활용 도구
- **Claude Code (Anthropic)**: 터미널 기반 AI 코딩 어시스턴트를 활용하여 프로젝트 전 과정에 참여

### AI를 활용한 작업 내역

#### 기존 소스 분석
- 기존 `gcp_pg_data_check`, `gcp_pg_data_sync` 파이프라인 소스 파일 전체를 AI가 읽고 분석
- 파이프라인 블록 간 데이터 흐름, DB 테이블 구조, 처리 로직을 AI가 정리하여 문서화
- 분석 결과를 `PROMPT.md`에 자동 반영하여 이후 개발의 컨텍스트로 활용

#### 로직 설계
- 현재 수동 처리 흐름의 문제점 파악 및 자동화 방안을 AI와 함께 설계
- 일 단위 재업로드의 비효율 문제 → 시간 단위 세분화 방식으로 전환하는 아이디어 구체화
- BQ 조회 비용 절감 방안(시간별 24회 → 날짜 범위 1회 조회) AI와 함께 도출
- GCS 백업 타이밍(업로드 완료 후 날짜 단위 1회) 결정 과정에 AI 분석 활용

#### 프로시저 분석 및 연동 방안 수립
- `export_2_gcs` BigQuery 프로시저를 AI가 분석하여 입력 파라미터, GCS 저장 경로 규칙 파악
- Python에서 BQ 프로시저를 호출하는 연동 방안(CALL 문) AI가 제안

#### 문서화
- `PROMPT.md` 작업 컨텍스트 문서를 AI가 지속적으로 유지/업데이트
- 본 보고서(`PROJECT_REPORT.md`) AI가 작성

### AI 활용 방식의 특징
- 단순 코드 생성이 아닌, **요구사항 분석 → 설계 → 검토 → 문서화** 전 과정에서 AI와 협업
- `PROMPT.md`를 통해 세션 간 컨텍스트를 유지하며 연속적인 작업 진행
- 개발자가 비즈니스 로직과 의사결정에 집중하고, AI가 소스 분석/문서화/코드 구조 설계를 담당하는 역할 분담

## 3. 전후 비교 설명

### 이전 (수동 처리)

| 항목 | 내용 |
|------|------|
| 비교 단위 | 일(Day) 단위 건수 비교 |
| 누락 감지 | Webhook 알림 수신 후 담당자가 직접 확인 |
| 시간별 비교 | 수동으로 해당 날짜 시간별 쿼리 실행 |
| 누락 대상 저장 | 수동으로 `gcp_pg_upload_error_hourly_ht` 에 입력 |
| 데이터 업로드 | 수동으로 파이프라인 실행 (파라미터 직접 입력) |
| GCS 백업 | 없음 (누락) |
| 처리 시간 | 알림 확인 + 수동 작업으로 수 시간 ~ 익일 처리 가능성 |
| 인적 오류 | 날짜/테이블 파라미터 입력 실수 가능성 존재 |

### 이후 (자동화 처리)

| 항목 | 내용 |
|------|------|
| 비교 단위 | 일 단위 비교 → 이상 감지 시 자동으로 시간 단위 세분화 |
| 누락 감지 | 자동으로 처리 대상 조회 (diff_cnt < -100) |
| 시간별 비교 | 자동으로 해당 날짜 전체 시간별 BQ/PG 비교 후 저장 |
| 누락 대상 저장 | 비교 후 `gcp_pg_upload_error_hourly_ht` 자동 UPSERT |
| 데이터 업로드 | 자동으로 차이 있는 시간대만 DELETE → INSERT |
| GCS 백업 | 업로드 완료 후 자동으로 BQ → GCS 백업 (`export_2_gcs` 프로시저 호출) |
| 처리 시간 | Mage.AI 파이프라인 스케줄에 따라 당일 자동 완료 |
| 인적 오류 | 파라미터 자동 생성으로 입력 실수 제거 |

### 처리 효율 비교

| 구분 | 이전 | 이후 |
|------|------|------|
| 담당자 개입 | 매 누락 발생 시 필수 | 장애 수준 이상 시에만 선택적 개입 |
| 재업로드 범위 | 누락 발생 일 전체 데이터 | 실제 차이가 발생한 시간대만 |
| BQ 조회 비용 | 시간별 N회 개별 쿼리 | 날짜 범위 1회 쿼리 후 Python 분리 |
| GCS 정합성 | BQ 데이터와 GCS 불일치 가능 | 업로드 후 GCS 자동 백업으로 정합성 유지 |

## 4. 기대 효과

### 운영 효율화
- **담당자 수동 작업 제거**: 누락 감지부터 업로드, GCS 백업까지 전 과정 자동화
  - 기존: 누락 1건 처리에 조회/확인/실행 포함 30분~1시간 소요
  - 자동화 후: 파이프라인 스케줄에 의해 자동 처리, 담당자 개입 최소화
- **야간/주말 누락 데이터 자동 복구**: 담당자 부재 시에도 누락 데이터 자동 처리 가능

### 데이터 품질 향상
- **시간별 정밀 복구**: 일 전체 재업로드에서 시간 단위 선별 업로드로 정확도 향상
  - 이상이 없는 시간대의 데이터를 불필요하게 삭제/재입력하는 위험 제거
- **신속한 복구**: 누락 발생 당일 자동 감지 및 복구로 데이터 지연 시간 최소화
- **일관된 처리**: 수동 작업의 인적 오류(파라미터 실수, 일부 테이블 누락 처리) 제거

### 비용 절감
- **BQ 조회 비용 최적화**: 시간별 24회 개별 쿼리 → 날짜 단위 1회 쿼리로 비용 절감
- **불필요한 전체 재업로드 제거**: 차이 없는 시간대 데이터 DELETE/INSERT 제거로 BQ DML 비용 절감

### 데이터 보안 및 복원력 향상
- **GCS 백업 자동화**: 수동 재업로드 후 GCS 백업이 누락되던 문제 해결
  - BQ와 GCS 데이터 정합성 항상 유지
  - GCS 백업 파일을 통한 장애 시 데이터 복원 가능
- **이력 관리**: `gcp_pg_upload_error_hourly_ht` 테이블에 처리 전/후 건수(`bq_cnt`, `after_work_bq_cnt`) 및 처리 여부(`work_yn`) 자동 기록

### 확장성
- 신규 사업장 또는 신규 데이터 종류 추가 시 메타 테이블(`clct_table_mt`) 등록만으로 자동 모니터링 범위 확장
- 표준화된 4단계 파이프라인 패턴을 조직 내 유사 시스템에 적용하여 전사적 데이터 품질 관리 체계 수립 가능
